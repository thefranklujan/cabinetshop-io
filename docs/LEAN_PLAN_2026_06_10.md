# CabinetShop.io — Lean Operating System: Audit and Implementation Plan

**Date:** 2026-06-10
**Status:** PLAN ONLY. No code in this pass. Build starts only on Frank's explicit approval.
**Constraint:** the pilot-ready foundation (commit `73a89a8`) must keep working untouched.

---

## 1. Executive summary

CabinetShop.io today is a solid **job tracker**: a 13-stage production board, per-job records, materials, cut lists, POs, schedule, invoices, and real multi-tenant RBAC. What it is not yet is a **lean operating system**: nothing in the app prevents the classic shop failure mode where a job rolls into production missing approvals, measurements, selections, or materials, and the waste shows up two weeks later as rework, idle benches, and angry phone calls.

The single highest-leverage move is not more modules. It is **gating flow on readiness**: a job cannot (or loudly should not) advance past key points until its approval gates and readiness checklist are green. That one mechanic operationalizes four of the five lean principles at once (value, value stream, flow, pull), and everything else in this plan (tasks, constraints, messaging, templates, dashboard) hangs off of it.

Recommended path: keep the current 13 board stages exactly as they are, add **approval gates + a job readiness checklist** as Phase 1, then tasks/constraints, then the message timeline, then templates, then the lean dashboard, and only then estimating (Phase 6) so that estimates are born inside the value stream instead of bolted on.

---

## 2. What CabinetShop.io already does well

- **The 13-stage board is already a value-stream map.** Quote → Design → Approved → Materials → Cut/CNC → Assembly → Sanding → Finish → QC → Delivery → Install → Punch List → Complete is a credible custom-shop flow, and the board with drag + per-stage WIP counts and dollar totals is genuine visual management.
- **WIP visibility.** Dashboard shows pipeline distribution and WIP value, the raw ingredients of bottleneck thinking.
- **Domain-correct records.** Projects carry wood species, finish, hardware, square feet, cabinet count, priority. Cut lists carry part/material/qty/dimensions with done-toggles (a real shop-floor artifact). Materials have reorder thresholds with low-stock surfaced on the dashboard.
- **Pull-adjacent primitives exist.** Reorder-at on materials is a kanban signal; PO statuses model the supplier loop.
- **Tenant isolation and RBAC are real and DB-enforced.** owner/admin/member/viewer roles behave correctly in UI and RLS. New lean tables can inherit a proven security pattern (`current_workspaces()`, `can_write_workspace()`, `is_workspace_admin()`, WITH CHECK).
- **Operational honesty.** No fake features; pricing/marketing copy does not overpromise estimating or billing.

## 3. Biggest lean/product gaps

1. **No gates.** Any job can be dragged from Quote straight to Install. Nothing encodes "do not start what is not ready," which is the root cause of rework and expediting.
2. **No definition of "ready for production."** Deposit, signed proposal, field measure, selections, drawings, materials: none are tracked as a checklist, so readiness lives in the owner's head.
3. **No tasks or owners.** The app tracks state, not work. Nobody can answer "what must happen today, by whom, to keep jobs flowing."
4. **No constraints view.** "Waiting on client / vendor / material / design" is invisible, so bottlenecks are discovered, not managed.
5. **No communication record.** Approvals, selections, and client conversations happen off-system; the job record cannot prove what was approved or when.
6. **No standard work.** Every job starts from zero; identical kitchens get inconsistent checklists (or none).
7. **Stage age is invisible.** A card can sit in Finish for three weeks and look identical to one that arrived an hour ago.
8. **Files are absent.** Drawings, signed proposals, and finish samples have no home (today this is a gap; v1 can start with links/notes, then Supabase Storage).

## 5. Lean value stream and the recommended v1 stage model

**Ideal custom-shop value stream (target):**
Lead/Inquiry → Estimate → Design → Client Approval → Field Measure → Engineering/Cut List → Materials Ordered → Materials Ready → Cut/CNC → Assembly → Sanding → Finish → QC → Delivery → Install → Punch List → Complete.

**Current app stages:** Quote, Design, Approved, Materials, Cut/CNC, Assembly, Sanding, Finish, QC, Delivery, Install, Punch List, Complete.

**Gap analysis:** the current model compresses Lead+Estimate into "Quote," compresses Client Approval+Field Measure+Engineering into "Approved," and compresses Ordered+Ready into "Materials." Those compressed steps are exactly where cabinet jobs die.

**Recommendation: do NOT expand the stage list in v1.** Reasons:

1. A 17-column board is physically unusable as visual management; lean boards stay readable.
2. Stages model **where the work physically is**; the missing steps are **states of information readiness**, which belong to gates and checklists, not columns.
3. Changing the Stage enum breaks live pilot data and the deferred "editable stages" promise. Hard constraint: do not break the pilot.

**v1 stage model = the existing 13 stages + gates layered on top:**

- "Quote" covers Lead/Inquiry/Estimate (Phase 6 adds estimate records beneath it).
- "Approved" is re-explained in UI copy as "Approved & Pre-Production": client approval, field measure, engineering all live here as gates that must close before the job leaves the stage.
- "Materials" covers Ordered→Ready via the PO rollup gate.
- Everything from Cut/CNC onward is already correct shop-floor flow.

The single most important boundary becomes **Approved → Materials**, which is the **Release to Production** line. That is where the readiness checklist gets enforced.

## 6. Approval gates design

One mechanism, nine instances. A gate is a named approval attached to a job, owned by someone, optionally due, that **warns or blocks** specific stage transitions until satisfied.

**Status values (all gates):** `not_started` → `in_progress` → `waiting_external` → `approved` | `declined` | `n_a` (n_a lets a vanity job skip the appliance gate without faking an approval; declined reopens to in_progress on revision).

**Enforcement modes:** `block` (cannot drag past the linked boundary; owner/admin may override with a logged reason) and `warn` (modal "this job is missing X, move anyway?", always allowed, logged). Default per gate below; configurable per workspace later (Settings, DB-backed per the self-sustainability rule).

| Gate | Blocks entry to | Mode | Default owner | Required fields | Files/comments |
|---|---|---|---|---|---|
| Estimate/quote approval | Design | block | owner | approved amount, client name, date | comment + link/file |
| Design approval | Materials (release line) | block | admin/designer | drawing reference, revision #, client sign-off note | file required (link in v1) |
| Finish approval | Materials | block | member assigned | finish name/sheen, sample confirmed y/n | photo/link |
| Hardware approval | Materials | warn | member assigned | hardware list confirmed | comment |
| Appliance/spec confirmation | Cut/CNC | warn | admin | spec sheet per appliance, or n_a | file/link |
| Field measure confirmation | Cut/CNC | block | member (field) | measure date, measured-by, notes | photos/link |
| Cut list release | Cut/CNC | block | admin | cut list reviewed-by, released date | auto-links cut list |
| Install readiness | Install | block | admin | site ready y/n, delivery confirmed, crew assigned note | comment |
| Punch completion | Complete | block | owner | all punch tasks closed, client sign-off note | comment |

Every gate transition writes an **activity row** (who, when, old→new, override reason if any). Overrides are visible on the dashboard, because frequent overrides are themselves a lean signal that the standard does not match reality.

## 7. Job readiness checklist ("Is this job ready for production?")

A per-job checklist, separate from gates (gates = approvals/decisions; checklist = facts). Items have `pending` / `done` / `n_a`, an owner, and a `required_for_release` flag. **Job is "Ready for production" when every required item is done/n_a and every release-line gate is approved.** Shown as a green/amber/red chip on the job row, job page, and board card.

Default items (template-driven, Phase 4 makes them editable):

1. Signed proposal on file (link/file)
2. Deposit received (auto-checks when an invoice marked "deposit" is Paid; manual override allowed)
3. Field measurements complete (auto from gate)
4. Drawings uploaded/linked
5. Finish selected (auto from gate)
6. Hardware selected (auto from gate)
7. Appliance/spec notes complete
8. Material ordered (auto: every job-linked PO ≥ Sent)
9. Material received (auto: every job-linked PO = Received)
10. Cut list ready (auto from cut-list-release gate)
11. Install date scheduled (auto: a Schedule event of type Install exists for the job)

Auto-derivation matters: 7 of 11 items compute from data the app already has or that gates produce, so the checklist stays true without ceremony.

## 8. Tasks and constraints system

**Task fields:** title; project_id (nullable for shop tasks); owner (workspace member); role/team tag (office/shop/field/design); due_date; status (`open`/`in_progress`/`done`/`canceled`); priority (Low/Normal/High/Rush, same enum as jobs); `is_blocker` flag; optional `waiting_on` (`client`/`design`/`material`/`vendor`/`install_date`/`internal`); optional stage link; comments (shared comments table).

**Views:** My Tasks (owner = me, due order), Job Tasks (on job page), Today (due/overdue), and:

**Constraints board** — the lean control tower. Columns are the waiting_on reasons:
Waiting on Client · Waiting on Design · Waiting on Material · Waiting on Vendor · Waiting on Install Date, plus two computed lanes: **Overdue approvals** (gates past due) and **Overdue tasks**. Each card shows job, what is waited on, owner, and **age in days**. Sort by age descending: the oldest constraint is today's first phone call. This view is the daily huddle screen.

## 9. Client/builder messaging (v1, deliberately small)

Not chat. A **per-job timeline** of typed entries:

- `internal_note` (team-only)
- `client_note` (marked client-visible; in v1 "client-visible" means it is part of the official record and will be exposed when a client portal exists; today the team reads it to the client or pastes it into email)
- `approval_request` (auto-created when a gate is set to waiting_external; records what was asked and when)
- `client_response` (logged by the team: "Maria approved slab white oak, per phone 6/14"; can resolve a gate)
- `system` (auto: stage moves, gate changes, override reasons — same activity stream)

One table, one timeline component on the job page, newest first, filter internal/external. **No email sending in v1** (optional notification pass later; the platform Resend integration exists but stays out of scope to protect the pilot). This gives the shop the thing it actually lacks: a defensible record of who approved what, when.

## 10. Standard work templates

Templates are workspace-owned, DB-backed, and editable in Settings (operational content is never hardcoded). A template defines: default tasks (with role + day offsets), checklist items (with required_for_release flags), and which gates apply with default modes/owners.

Seed templates shipped as workspace defaults on first use:

| Template | Notable differences |
|---|---|
| Kitchen job | Full gate set; appliance gate = block; longest task list |
| Bath/vanity job | Appliance gate n_a by default; shorter finish loop |
| Built-in/closet job | No appliance gate; field measure emphasized |
| Commercial/millwork job | Adds submittal-approval task pack; hardware gate = block; builder (not homeowner) wording |
| Install prep | Task-pack only (no gates): site check, protection, hardware kits, touch-up kit |
| Punch list | Task-pack: walkthrough, punch items as tasks, client sign-off task |
| Finish approval | Mini-pack: sample made → sample sent → client confirmed (drives finish gate) |

Applying a template to a job instantiates rows (copies, not references), so editing a template never rewrites history on live jobs.

## 11. Dashboard and visual management (concept only, no visual redesign)

Reframe the dashboard as the shop's andon board. Same card/section design language as today.

**Row 1 — flow health:** Jobs Ready for Production (green, count + $) · Jobs NOT Ready (red, count + worst offender) · Blocked Jobs (constraint count) · Overdue Approvals.
**Row 2 — this week:** Installs due this week · Material shortages (existing low-stock + any job waiting on material) · Open punch items.
**Row 3 — flow analytics:** WIP by stage (existing bars) annotated with **aging** (avg days-in-stage per column; worst cell highlighted = bottleneck candidate) and a simple **bottleneck callout** ("Finish holds 5 jobs / $62k, avg 9 days").
**Board changes:** stage-age badge on each card (days in current stage), red "blocked" corner flag, readiness chip on pre-production cards.

All counts are clickable into the filtered list/constraints view. No new chart library; bars and badges only.

## 12. Data model proposal (Supabase, additive only)

All tables: `id uuid pk default gen_random_uuid()`, `workspace_id uuid not null references workspaces(id) on delete cascade`, `created_at timestamptz default now()`. All follow the proven RLS pattern: SELECT via `workspace_id in (select current_workspaces())`; INSERT/UPDATE via `can_write_workspace(workspace_id)` with WITH CHECK pinning workspace_id; DELETE via `is_workspace_admin(workspace_id)`. Viewer = read-only automatically. No existing table is altered except small additive columns noted.

```
gates
  project_id uuid not null -> projects
  gate_key text not null            -- 'estimate_approval', 'design_approval', ...
  status text not null default 'not_started'
    check (status in ('not_started','in_progress','waiting_external','approved','declined','n_a'))
  mode text not null default 'warn' check (mode in ('block','warn'))
  blocks_stage text                 -- stage name whose ENTRY this gates
  owner_user_id uuid null -> auth.users (validated app-side against workspace_members)
  due_date date null
  fields jsonb default '{}'         -- gate-specific required fields
  resolved_at timestamptz, resolved_by uuid
  unique (project_id, gate_key)

checklist_items
  project_id uuid not null -> projects
  item_key text not null            -- 'signed_proposal', 'deposit', ...
  label text not null
  status text not null default 'pending' check (status in ('pending','done','n_a'))
  required_for_release boolean not null default true
  auto_source text null             -- 'gate:finish_approval' | 'po_received' | 'invoice_deposit' | 'schedule_install' | null(manual)
  owner_user_id uuid null
  done_at timestamptz, done_by uuid
  unique (project_id, item_key)

tasks
  project_id uuid null -> projects
  title text not null
  owner_user_id uuid null
  team text null check (team in ('office','shop','field','design'))
  due_date date null
  status text not null default 'open' check (status in ('open','in_progress','done','canceled'))
  priority text not null default 'Normal'
  is_blocker boolean not null default false
  waiting_on text null check (waiting_on in ('client','design','material','vendor','install_date','internal'))
  stage text null
  template_item_id uuid null

messages                            -- job timeline (comments + communication log)
  project_id uuid not null -> projects
  kind text not null check (kind in ('internal_note','client_note','approval_request','client_response','system'))
  body text not null
  gate_id uuid null -> gates        -- when tied to an approval
  task_id uuid null -> tasks
  author_user_id uuid not null
  -- viewer note: viewers can read; only can_write_workspace may insert; nobody updates (append-only); admin may delete

activity                            -- immutable audit/timeline
  project_id uuid not null -> projects
  actor_user_id uuid not null
  verb text not null                -- 'stage_moved','gate_approved','gate_overridden','task_done',...
  detail jsonb default '{}'         -- {from,to,reason,...}
  -- INSERT via trigger/app only; SELECT workspace-scoped; no UPDATE/DELETE policies (append-only)

job_templates
  name text not null, job_type text not null, is_default boolean default false
template_items
  template_id uuid not null -> job_templates
  kind text not null check (kind in ('task','checklist','gate'))
  payload jsonb not null            -- task title/team/offset | checklist label/required | gate key/mode/owner-role
  sort int not null default 0
  -- templates: SELECT all members; INSERT/UPDATE/DELETE admin-only (is_workspace_admin)

additive columns on existing tables
  projects: released_to_production_at timestamptz null, template_id uuid null
  invoices: is_deposit boolean not null default false
```

Constraints view needs no table: it is a query over tasks (is_blocker/waiting_on/overdue) + gates (due_date past, waiting_external). RLS notes: member can write gates/checklist/tasks/messages but not delete (matches current shop rules); gate `mode='block'` override allowed only when `is_workspace_admin()` and always writes an `activity` row with a reason; `activity` and `messages` are append-only by policy so the approval record is trustworthy.

## 13. Implementation roadmap

**Phase 1 — Readiness + approval gates (the lean core)**
Goal: no job enters production missing approvals; readiness visible everywhere.
User value: kills the #1 waste source (premature starts, rework).
Data: `gates`, `checklist_items`, `activity`, additive project/invoice columns; one migration.
UI: Readiness panel on job page (checklist + gates), readiness chip on board/list cards, warn/block modal on board drag, auto-derivation wiring (POs, deposit invoice, install event, cut list).
Risk: **medium** — touches board drag (the pilot's showpiece). Mitigate: gates default to `warn` for existing jobs; `block` only for jobs created after the feature ships; zero schema changes to existing columns.
Verify: drag a gated job (warn + block + admin override w/ reason), checklist auto-derivation against live PO/invoice/schedule data, RLS tx-simulation per role (the proven rollback method), full smoke-test regression of board persistence, `npm run check`, live deploy + browser pass.

**Phase 2 — Tasks, constraints, daily shop focus**
Goal: the shop knows what to do today and what is stuck.
User value: replaces the whiteboard/sticky notes; aging makes bottlenecks calls obvious.
Data: `tasks` (+ messages.task_id ready); member emails surfaced on Team (prereq, small).
UI: My Tasks + job tasks tab + Constraints board + dashboard "Blocked jobs" card.
Risk: low (pure addition).
Verify: per-role CRUD matrix, blocker → constraints lane → dashboard count round-trip, overdue computation, regression smoke.

**Phase 3 — Client/builder message timeline**
Goal: defensible record of asks, approvals, and responses per job.
User value: ends "but you said..." disputes; approvals get timestamps.
Data: `messages` (append-only RLS).
UI: timeline on job page, internal/external filter, auto `approval_request`/`system` entries from gates.
Risk: low. Explicitly NOT building: email sending, client login.
Verify: append-only enforcement (update/delete denied per role), gate → auto-message linkage, viewer read-only.

**Phase 4 — Standard work templates**
Goal: every job starts with the right gates/checklist/tasks for its type.
User value: consistency without thinking; onboarding a new PM becomes trivial.
Data: `job_templates`, `template_items`, seed defaults per workspace.
UI: template picker on New Project, template manager in Settings (admin-only).
Risk: low-medium (instantiation correctness). Copies not references.
Verify: create job from each seed template, edit template → confirm old jobs unchanged, role matrix on Settings manager.

**Phase 5 — Lean dashboard + reporting**
Goal: dashboard becomes the andon board (Section 11).
User value: 10-second morning read: what is ready, stuck, aging, due.
Data: none new; queries + `activity`-derived stage ages (stage_moved events make aging computable without new columns).
UI: dashboard rows 1-3, stage-age badges on board cards, clickable drill-downs.
Risk: low; read-only.
Verify: cross-check every count against module pages, empty-workspace rendering, perf on realistic data volume.

**Phase 6 — Estimating/quoting in the value stream**
Goal: estimates live under the Quote stage and flow into the job (the first paid feature).
User value: one system from inquiry to invoice; estimate approval gate becomes real data.
Data: `estimates`, `estimate_lines` (materials/labor/margin/tax/deposit), link to projects; deposit invoice generated from accepted estimate.
UI: estimate builder, proposal output (print/PDF later), accept → auto-creates deposit invoice + flips estimate gate.
Risk: highest (money math, proposal output); gated on pilot feedback per standing decision.
Verify: penny-exact totals fixtures, accept-flow E2E, RLS matrix, full regression.

## 14. First build-pass recommendation

**Phase 1 only: readiness checklist + approval gates.** One migration, one job-page panel, one board-drag modal, derivations from existing data. It is the smallest change that converts the tracker into a lean system, it is additive (pilot-safe via warn-mode for existing jobs), and every later phase depends on its tables. Phases 1+2 together would also fit a single focused build pass if Frank wants the daily-focus payoff immediately; anything more in one pass risks the pilot foundation.

## 15. What should wait until after pilot feedback

- Phase 6 estimating (already locked: wait for pilot signal)
- Email notifications and any client-facing portal/login
- File uploads via Supabase Storage (v1 uses links/notes; storage adds bucket policy surface)
- Editable production stages (existing deferred promise; templates partially cover the need)
- Visual redesign of any kind; reports beyond the dashboard cards
- Billing/Stripe (explicitly out), integrations (QuickBooks etc.), native app
- Material-to-job allocation/consumption tracking (real lean value, but heavy; needs pilot validation that shops will maintain inventory at all)
